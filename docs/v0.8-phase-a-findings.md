# v0.8 Phase A findings: static RE for mbox 0x204 path

## TL;DR

**libocxbpeng.so.2.3 exports the full Class3 connected-messaging stack
under the `OC_bp*` symbol family — including `OC_bpOpenClass3Conn`,
`OC_bpSendClass3Request`, `OC_bpCloseClass3Conn`, and
`OC_bpRegisterConnectionObj` / `OC_bpUnregisterConnectionObj`.** All
five symbols are real exported entry points, are NOT in libocxbpapi's
import table (so they were never reachable through the standard
`OCXcip_*` surface that bpServer uses), and their call chains pass
through libocxbpeng's internal worker threads (`oc_UcmTask`,
`cm_ObjectTask`, `APX_AsrTask`) which own the only two
mbox-0x204-posting sites in the binary: `apx_LinkRequest` (0x0018a8a8)
and `apx_AppResponse` (0x0018a6cc). **The OCXCN_* family that v0.7
dead-ended on is a libocxbpapi-side shim only; libocxbpeng has the
real implementation and does NOT depend on OCXCN_***. Recommended path
for v0.8: dlsym those five symbols from libocxbpeng directly,
bypassing libocxbpapi entirely for the Class3 path.

## Kernel module (eecbpmod.ko)

The kernel module is a thin user-mode passthrough.  Functional surface:

- Two miscdevs registered via `ocxbp_probe` (0x001019d0): `shram_miscdev`
  and `cbregs_miscdev`.  Userland opens `/dev/ocx_shram` and
  `/dev/ocx_cbregs` (string refs at libocxbpeng `0x002722a0` /
  `0x002722b0`).
- `ocxbp_mmap` (0x00100190) is the primary userland interface — maps
  cbregs (BAR 0x30) or shram (BAR 0x48) into the caller's address
  space.  Single-mapper exclusion on shram
  (`shram_map_count > 0 → -EBUSY`); no such limit on cbregs.
- `ocxbp_read` (0x00100164) returns -ENOTSUP.  All bulk transfer is
  through the mmap'd regions plus a handful of ioctls.

Ioctl numbers from `ocxbp_ioctl` (0x00100810):

| ioctl       | Direction | Purpose                                      |
| ----------- | --------- | -------------------------------------------- |
| 0x40084304  | write     | **`*(uint32_t*)(cbregs + offset) = value`** (raw register write) |
| 0xc0084303  | read+write| read 32-bit register at offset               |
| 0x80384302  | read      | get device info (size 0x38)                  |
| 0x805c4314  | read      | get diagnostics (size 0x5c)                  |
| 0x80044309/0a/0b/0c/24 | read | dequeue CB from per-type SAM queue       |
| 0x42084319  | r+w       | write SHRAM block (uses cb-write helpers)    |
| 0xc2084318  | r+w       | read SHRAM block                             |
| 0x4008431c  | write     | enqueue work_struct (system_wq)              |
| 0x4010431d  | r+w       | wait on completion w/timeout (200 ms cap)    |
| 0x40014313/15 | write  | set/clear ASIC interrupt-enable bit          |
| 0x80044301  | read      | get ASIC build/version (returns 0x40001)     |
| 0x8004430a  | read      | misc state                                   |
| 0x8004431a, 0x8004431e | read | counters / flags                       |
| 0x8008431b  | r+w       | allocate work_struct                         |
| 0xc012430d  | r+w       | EEPROM read (`SAMma_BackplaneEEPROMread`)    |
| 0x430f, 0x4310 | -      | reset ASIC / set fault flag                  |

**There is no ioctl for "post to mailbox N".**  The kernel doesn't
distinguish mboxes — userland posts by directly writing
`cbregs[mbox_offset/4] = cb_off | 0x400000` via the mmap'd region,
or by calling ioctl `0x40084304` with `offset = mbox_offset` to do
the same write through the kernel.  Libocxbpeng's `apx_safe_reg_write`
(0x0018f0ec) uses the ioctl variant; the inline mbox post primitive
`APX_SendRequest` (0x0018cb60) goes through the mmap'd region.

The only place where the kernel module itself posts to a mailbox is
inside `ocxbp_interrupt` (0x001021c0) where `APX_SendRequest(p_info->cbregs + 0x208, cb)`
is called for CB types 5–13 that need a "free list" callback —
that's mbox 0x208 (MBOX_UNCONN), not 0x204.  The IRQ handler also
dispatches received CBs onto per-type SAM CB queues
(`SAMputCbQ_isra_0(plVar8, cb_off)`), which userland workers pull
from via the read ioctls.

So the kernel sets the bar at *zero*: posting to mbox 0x204 is one
mmap'd 32-bit store; no kernel API to call.  Anything more is purely
userland orchestration.

## libocxbpeng.so.2.3 — every mbox 0x204 site

Static analysis (byte-pattern search for AArch64 `add x0, ?, #0x204`
across `.text`, plus decompile verification) finds exactly two
functions that pass `pApex + 0x204` as the first argument to the
generic mbox-post primitive `APX_SendRequest` (aliased internally as
PLT thunk `func_0x00187ce0` resolving to the library's own export at
0x0018cb60):

| Function          | Address       | What it does                                                                                          |
| ----------------- | ------------- | ----------------------------------------------------------------------------------------------------- |
| `apx_AppResponse` | `0x0018a6cc`  | When a Class3 reply CB arrives with matching sequence, posts either the primary (param+0x38) or secondary (param+0x40) CB to mbox 0x204. |
| `apx_LinkRequest` | `0x0018a8a8`  | When a Class3 transport completion arrives with no error and a non-NULL secondary CB whose RecvSeq matches the expected RecvSeq, posts (param+0x40) to mbox 0x204. |

Both functions are CB-completion handlers — they fire from a worker
task that pulls a completed CB off a SAM queue, and they re-post a
second CB (the "consume" / "transport" CB) into the connected mailbox
to keep the Class3 transport cycle going.  This matches the sibling
`apex2_cip_connection.c::cip_connected_send_large` (lines 1411–1413,
1477) behavior: the producer CB lands a response and the consumer CB
needs to be re-posted via `_mailbox_post(0x0204, rcb)` to receive the
next packet.

The other six byte-pattern hits for `add x0, ?, #0x204` are not
mailbox posts — they are struct offset computations against
non-`pApex` bases (verified by reading the surrounding decompile):

| Address      | Containing function   | Use                                  |
| ------------ | --------------------- | ------------------------------------ |
| `0x0018a734` | `apx_AppResponse`     | (additional code path in same fn, also posts to 0x204) |
| `0x0018a7d4` | `apx_AppResponse` end | epilogue/exit, struct offset         |
| `0x0018aa08` | `apx_LinkRequest`     | the post call site                   |
| `0x0018ba5c` | `apx_ChassisInit`     | struct offset (chassis state)        |
| `0x0018cf6c` | `APX_PutOnHostFreeList`/Synch helper | struct offset           |
| `0x00199834` | (unnamed, deep in `.text`) | struct offset, not pApex base   |
| `0x0019b838` | `OC_bpImmediateOutput` area | struct offset                  |
| `0x0019b920` | `OC_bpImmediateOutput` area | struct offset                  |

The mbox-0x204 sites are concentrated entirely in the asynchronous
response handler path (`apx_AppResponse`, `apx_LinkRequest`) — they
are NOT called synchronously from any exported entry point.  They
fire only as part of an in-flight Class3 conversation.

### Why search_instructions returned only PLT matches

`mcp__ghidra-mcp__search_instructions` on `libocxbpeng.so.2.3` for
operand `0x204` returned 0 matches but `instructions_scanned=6712`
which is the count of PLT-region instructions only (the `.plt`
covers `0x001836c0–0x00189f9f`, ~6712 instructions of 4 bytes).
The tool isn't scanning the main `.text` block (`0x00189fa0–
0x0026ba4b`).  All findings above were obtained via
`search_byte_patterns` with masked AArch64 instruction encodings
and cross-referencing decompile output.  Treat the search_instructions
tool as PLT-only on this image.

There is **no `OCXcip_*` export that reaches mbox 0x204 without
going through `OCXCN_*`**.  Confirmed by enumerating all 60-odd
exports and tracing each through its decompile.  `OCXcip_TxRxOpenConn`
and `OCXcip_TxRxMsg` are the only Class3 paths and both require
OCXCN_*.

`OCXcip_ReadConnected` / `OCXcip_WriteConnected` are a red herring:
their name suggests "connected messaging" but they wrap
`OC_bpReadConnected` / `OC_bpWriteConnected` in libocxbpeng, which
operate on the *local assembly-object* state cache only.  They do
not generate any wire traffic, never call into the CM worker, and
never post to any mailbox.  Verified by decompiling
`OC_bpReadConnected` (0x0019b988): it just memcpy's from
`OC_sBpAssemblyObj` under `OC_AssmCritS` lock.

## bpServer — practical use of mbox 0x204

bpServer is statically linked to libocxbpapi (via the
`OCXcip_TxRx*` family) and contains:

- **Zero** instances of the byte `0x204` or `0x0204` as a 32-bit
  immediate (verified with `search_byte_patterns`).
- **Zero** imports from libocxbpeng — every connection-relevant
  symbol bpServer pulls is `OCXcip_*` from libocxbpapi.
- **No dlsym/dlopen** in its import list — bpServer cannot late-bind
  symbols.

bpServer's Class3 path is therefore strictly:

```
bpServer
  → OCXcip_TxRxOpenConn / TxRxMsg / TxRxCloseConn (libocxbpapi)
    → OCXCN_OpenClass3Connection / SendClass3Request / CloseClass3Connection (MISSING libocxcn.so)
      → would call OC_bpOpenClass3Conn / SendClass3Request / CloseClass3Conn (libocxbpeng, present)
```

**Bottom line: bpServer is also broken for Class3 messaging on this
image.**  It can only use mbox 0x200 (UCMM via `OCXcip_MessageSend`)
and the local assembly-object read/write helpers.  The shipped
firmware image is genuinely missing the OCXCN_* shim layer.

This is consistent with the v0.7.0 observation that
`OCXcip_TxRxOpenConn` returns `0x1001` — a "feature not implemented"
or "symbol not resolved" code coming from the dynamic linker /
PLT-resolution stub when OCXCN_* can't be bound.

### Static evidence that libocxbpeng owns the full Class3 stack

1. **The CM worker is in libocxbpeng**: `cm_ObjectTask` at
   `0x00192888` dequeues from `CM_xQid` and dispatches on opcode.
   Opcodes 0x31 (open), 0x32 (close), 0x2f/0x30 (cm state), 0x20
   (UCMM) are all handled in-library.
2. **The connection-state functions are in libocxbpeng**:
   `cm_OpenConnection` (0x190058), `cm_CloseConnection` (0x191334),
   `cm_ParseFwdOpen` (0x191c24), `cm_ProcessFwdOpen` (0x19227c),
   `cm_ParseLgFwdOpen` (0x191f64), `cm_IssueFwdOpen` (0x18fb98),
   `cm_IssueFwdReply` (0x190de4), `cm_FindConnection` (0x190bb0),
   plus 20-odd more `cm_*` symbols.  This is the complete CIP
   Class3 state machine.
3. **The UCMM worker is in libocxbpeng**: `oc_UcmTask` at
   `0x0019d934` handles opcodes 0x20 (UCMM), 0x31 (open conn —
   forwards to CM worker), 0x32 (close conn), 0x33 (special).
4. **`OC_bpOpenClass3Conn` (0x0019d094) directly queues opcode
   0x31 to `OC_Ucm_xQid`** (`PTR_OC_Ucm_xQid_002bf718`) —
   confirmed in decompile, line `*(undefined4 *)(lVar7 + 8) = 0x31;`
   and `*(undefined8 *)(lVar7 + 0x10) = *(undefined8 *)PTR_OC_Ucm_xQid_002bf718;`.
5. **The mbox-0x204 sites are in the response-handling path**:
   `apx_AppResponse` (0x0018a6cc) and `apx_LinkRequest`
   (0x0018a8a8), driven by IRQ → SAM CB queue → worker task → CB
   dispatch table.  No external symbol involved.

The OCXCN_* shim presumably exists to translate the `OCXcip_*`
multi-handle/multi-slot API (libocxbpapi's `Cl3State` array at
`0x130068` with 0x130-byte entries) into the simpler
single-instance `OC_bp*Class3*` API.  We can simply do that
translation in our own SDK and skip OCXCN_* and libocxbpapi
entirely.

## Recommendation

**Use libocxbpeng directly — dlsym the five `OC_bp*Class3*` /
`OC_bp*ConnectionObj` symbols.**  v0.8 wires up to:

- `OC_bpRegisterConnectionObj(handle, connId_out, connObj_in, recvFn, recvCtx)`
  — installs the inbound-message callback (server-side, for our
  consume CB).
- `OC_bpOpenClass3Conn(handle, fwd_open_params, slot, conn_id_out,
   recvCb, ...)`  — runs the Forward Open through the CM worker.
- `OC_bpSendClass3Request(handle, conn_id, request_data, len)`  —
  synchronous Class3 send; internally builds a transport CB and
  triggers the mbox-0x204-bearing response-handling loop.
- `OC_bpCloseClass3Conn(handle, conn_id, ...)`  — Forward Close.
- `OC_bpUnregisterConnectionObj(handle, conn_id, ...)`  — release.

This is a libocxbpeng-only path; libocxbpapi is bypassed for the
Class3 family.  The library already does all the heavy lifting:

- CB allocation (`APX_ExDataBlkMalloc` etc.)
- SHRAM layout (libocxbpeng owns the `pSharedRAM` mmap and the
  CB free-list, so we coexist with bpServer's allocations
  *via the same library state* — no two-userland race because
  there's only one copy of the engine state)
- IRQ-driven response routing (the `APX_AsrTask` thread is started
  by `OC_bpStartup` and runs for the life of the process)
- mbox 0x204 / 0x200 / 0x208 posting (handled internally)
- pthread sync primitives (callback delivery is on the engine's
  own thread; we wrap with our own condvar for caller wait)

What v0.8 has to build on top:

1. **Process-attach decision**: `OC_bpStartup` must be called
   before any other `OC_bp*` function, and it opens
   `/dev/ocx_cbregs` + `/dev/ocx_shram`.  bpServer already called
   it.  Calling it a second time from our process *may* work
   (each process gets its own file descriptors and mmaps), but
   there is now *two* userland processes with the engine state.
   They will fight over CB IDs unless one of:
   - (a) we link libocxbpeng but *don't* call `OC_bpStartup` and
     instead splice into the existing CB pool — likely doesn't
     work because the worker threads run inside the process that
     calls Startup.
   - (b) we run in a forked child of bpServer — not possible from
     a sidecar container.
   - (c) **we accept that we're starting our own engine instance
     in our process**.  This means we own our own CB free list,
     our own worker threads, our own mmap of shared SHRAM.  The
     ASIC's CB lookup table is global to the physical chip, so
     two simultaneous engine instances would step on each other
     unless we carve out a CB-ID range.
   - (d) we go SHRAM-direct (the apex2_lib.h approach in
     sibling), allocating from a pre-arranged CB-ID range.
   This is a v0.8 Phase B/C dynamic-RE question — see below.

2. **Connection-object callback contract**: we need to figure out
   the exact recvFn signature — `OC_bpRegisterConnectionObj` is
   exported but undocumented externally.  Its decompile at
   `0x0019ce8c` shows it stores `param_4` and `param_5` (recvFn
   and recvCtx) plus `param_3` (a uint32_t — likely the connection
   class/instance/attr triple as in the standard CIP CM AssyObj
   API).  Phase B will exercise it with stubs at runtime.

3. **Forward Open parameters**: `OC_bpOpenClass3Conn`'s decompile
   at `0x0019d094` shows it takes a Forward Open parameter block
   at `param_2` — a struct with at least 0x24 bytes (T→O conn ID,
   O→T conn ID, transport key bytes, timeout ticks, RPI etc.).
   The exact layout needs an offset-by-offset RE pass against
   `cm_OpenConnection` (which reads it field by field) and
   `cm_ParseFwdOpen` — see appendix below for offsets we already
   have.

**Fallback if Phase B dynamic RE shows `OC_bpOpenClass3Conn` does
not actually work from a non-bpServer process** (e.g., the engine
state has process-affinity baked in via static globals like
`PTR_pApex_002bf750` and `PTR_pSharedRAM_002bff48` not being
re-initialized correctly from a second `OC_bpStartup`): fall
through to SHRAM-direct, using `apex_init_cb` / `apex_pop_cb` /
`apex_mailbox_post` from the sibling's `apex2_lib.h`.  That
*does* work standalone but races bpServer over CB IDs unless we
pre-arrange a range.
