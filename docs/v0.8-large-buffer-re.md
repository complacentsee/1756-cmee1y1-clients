# v0.8 RE plan: large-buffer connected messaging via mbox 0x204

> **Status (2026-05-22): NOT POSSIBLE on cm1756 from a rootless / non-
> privileged container.** Phase A static RE established the technical
> path (libocxbpeng directly exports the full Class3 stack under
> `OC_bp*Class3*`) but Phase A also confirmed that the precondition —
> a userland mmap of `/dev/ocx_shram` — is gated by a global
> single-mapper exclusion in `eecbpmod.ko` that bpServer holds for
> its entire process lifetime. We run as the non-privileged `bpapi`
> user (uid 1007 in rootless Docker) and cannot stop bpServer.
> Detailed findings + verdict at the bottom of this file; the
> [Phase A static-RE report](v0.8-phase-a-findings.md) has the
> complete reachability matrix.

## Why this exists

v0.7.0 ships functional `bp_client_txrx_open` / `txrx_msg` /
`txrx_close` by routing through `bp_client_message_send`
(chip mailbox 0x200, UCMM transport).  That gives callers the
PLC-side connection lifecycle — useful for asserting reachability,
explicit lifetime, and timing context — but **same envelope size
as MessageSend** (~500 B request, single-frame response).

The real benefit of CIP Class 3 connected messaging on this chip
family is 4002-byte packets over **mbox 0x204** with pre-registered
transport / consume CBs.  The sibling `historianupdate` apex2d
daemon uses this path heavily ([`apex2_cip_connection.c`](../../historianupdate/driver/apex2/daemon/apex2_cip_connection.c)
`::cip_connected_send_large`,`::cip_transport_send`).  That path
is not reachable through `OCXcip_MessageSend`.  v0.8 explores
whether the SDK can drive it directly.

## What we know going in

From the v0.7.0 Phase 1 RE pass and prior memory:

- **The kernel module on cm1756 is name-compatible with
  historianupdate's apex2d expectations.**  `eecbpmod.ko` exposes
  `/dev/ocx_cbregs` (major 10 minor 121) and `/dev/ocx_shram`
  (major 10 minor 122).  Sibling code opens the same paths.  Both
  modules are ASEM-shipped, both wrap the same APex2 ASIC family.
- **Sibling uses two distinct mailboxes:**
  - `0x200` (MBOX_LOOPBACK) — UCMM and degenerate "connected"
    sends; this is the path our SDK already drives via
    `OCXcip_MessageSend`.
  - `0x204` — connected-data transport.  Posted via
    `apex_mailbox_post(0x0204, cb)` where `cb` is a transport CB
    whose buffer pointer (CB+0x2C) points to a 4 KB SHRAM chunk.
- **Forward Open sets up the transport CBs**, not the wire
  framing.  Sibling `arm_transport_cbs()` allocates a 4 KB+4 KB
  SHRAM arena chunk and a consume CB (T→O receive) before sending
  the LFO.  The consume CB's dynamically-assigned ID is encoded
  as the low 16 bits of the T→O conn ID in the LFO body, so the
  PLC's CBS RX routing can index back into `CB_LOOKUP[wire_tag]`
  to find this CB.
- **Response routing is asynchronous.**  For mbox 0x204 sends the
  response arrives via the MAASR queue → consume CB, not by
  polling the produce CB's status.  Sibling registers per-conn
  pthread condvars (`resp_mtx` / `resp_cond` / `resp_ready`) and
  the IRQ handler signals them when responses land.

## RE questions to answer

1. **Can we open `/dev/ocx_cbregs` and `/dev/ocx_shram` from a
   non-root process?**  bpServer runs as uid 1007 / gid 1007
   (`bpapi`); our containers run as that uid via the rootless
   Docker config.  Sibling's apex2d may run as root.  If we need
   root we have a deployment problem — though rootless Docker is
   the only configuration knob (per [`docker-only`](../../.claude/projects/...memory pointer)).
2. **Can we share SHRAM with bpServer safely?**  bpServer already
   has the SHRAM mapped and is allocating CBs.  Two independent
   userland processes both calling `apex_pop_cb` against the same
   free list would race.  Either:
   - We pre-arrange a CB ID range with bpServer (out of scope —
     ASEM-shipped).
   - We never call `apex_pop_cb` and instead use a CB ID
     pre-allocated by some other mechanism (Forward Open response
     tells us our O→T conn ID; is the low 16 our CB?).
   - We use the `OCXcip_*` private library's connected-send
     internals directly via `dlsym`.  RE'd previously: those
     thunks resolve to `OCXCN_*` which is missing.  Dead end.
3. **Is there a private `libocxbpeng.so.2.3` entry point we
   missed?**  The `bp_client_txrx_*` path goes to
   `OCXcip_TxRx*` which goes to the missing `OCXCN_*`.  But
   `libocxbpeng` itself does the connected work in apex2d's
   layer-equivalent — maybe there's an unexported symbol that
   does the right thing.  Worth a fresh Ghidra pass with the
   knowledge gained from the sibling.
4. **Could we use a separate, parallel ASIC interface?**  i.e.
   skip libocxbpeng entirely and link historianupdate's apex2_lib.h
   directly.  Sibling's `apex_init_cb` / `apex_alloc_cb` /
   `apex_mailbox_post` are header-only.  But this assumes
   bpServer doesn't already own the CB space we'd be touching.

## Phase plan (TBD)

- **0.8 Phase A**: Static RE — re-pass `libocxbpeng.so.2.3` and
  `libocxbpapi.so.2.3` looking for any exported symbol that
  reaches mbox 0x204.  Confirm `OCXCN_*` is the only path, then
  the choice is "go around libocxbpeng" or "give up".
- **0.8 Phase B**: Dynamic RE — strace bpServer during an OCX
  shim Forward_Open + connected read sequence (the OCX dump
  referenced in sibling code at "CB[70]") to confirm what mbox
  it actually posts to and what CBs it sets up.
- **0.8 Phase C**: Decision — implement a SHRAM-direct connected
  transport in the SDK, mirroring apex2d's `cip_connected_send_large`,
  OR document this as permanently blocked and ship v0.8 with
  only QoL improvements.
- **0.8 Phase D** (if C is "go"): SDK changes.  Connection state
  expands to track transport CBs.  TxRxMsg gets a size-based
  dispatch: ≤500 B → MessageSend path (v0.7), >500 B → mbox 0x204
  path.  Response handling switches to async via the kernel's
  IRQ → SHRAM signaling primitive (whatever that turns out to be).

## What v0.7.0 leaves on the table

- **Throughput**: same as MessageSend.  ~500 B / 0.5–1 ms is fine
  for occasional tag reads; not fine for bulk scanning.
- **Tags larger than ~120 DINTs in one read**: not supported
  through TxRxMsg in v0.7.0; caller must chunk.  Sibling's
  4 KB transport handles ~1000 DINTs in one shot.
- **Response reassembly**: not needed at v0.7.0 envelope size,
  so the SDK has no multi-frame reassembly logic.  Adding it is
  part of v0.8 Phase D.

## Phase A verdict (2026-05-22)

Phase A static RE answered the technical question definitively — and
then a separate operational question made the answer irrelevant.

### What Phase A found

`libocxbpeng.so.2.3` **directly exports the full Class3 connected-
messaging stack** under the `OC_bp*` symbol family — see the
[Phase A report](v0.8-phase-a-findings.md) for the full reachability
matrix.  The key exports:

```
OC_bpRegisterConnectionObj   — install inbound-message callback
OC_bpOpenClass3Conn          — Forward_Open via the engine's CM worker
OC_bpSendClass3Request       — synchronous Class3 send
OC_bpCloseClass3Conn         — Forward_Close
OC_bpUnregisterConnectionObj — release
```

These bypass the missing `OCXCN_*` shim entirely.  libocxbpeng has
no `OCXCN_*` imports (only libc / pthread / libstdc++ / gpiod), so
the connected stack inside libocxbpeng is fully self-contained.
The mbox-0x204 posting sites (`apx_LinkRequest` @ `0x0018a8a8` and
`apx_AppResponse` @ `0x0018a6cc`) live inside libocxbpeng and are
driven by its own `APX_AsrTask` IRQ-handler thread.

As a side discovery: **`bpServer` is also broken for Class3** on
this image — it pulls only `OCXcip_*` from libocxbpapi, which
dead-ends at the missing `OCXCN_*`.  Class3 connected messaging
hasn't worked from any userland on cm1756 since the image shipped.

So at the library level v0.8 would just dlsym those five `OC_bp*`
symbols and we'd be done.

### Why we can't ship it

`OC_bpStartup` (the engine's per-process init function, required
before any other `OC_bp*` call) needs a **userland mmap of
`/dev/ocx_shram`**.  The kernel module `eecbpmod.ko` enforces a
**global single-mapper exclusion** on that device — directly
verified in the `ocxbp_mmap` decompile at `0x00100190`:

```c
if (_shram_miscdev == iVar3) {
    if (0 < shram_map_count) {
        return 0xfffffff0;       // -EBUSY
    }
    shram_map_pid = ...;
    shram_map_count = shram_map_count + 1;
    ...
}
```

`shram_map_count` is a process-independent global atomic.  bpServer
holds the mapping for its entire lifetime, and we run as the
non-privileged `bpapi` user (uid 1007) in rootless Docker — we
cannot stop, suspend, or co-opt bpServer.

### The ioctl back-door doesn't help either

The kernel module exposes ioctls that do SHRAM I/O on the caller's
behalf via its own mapping, no user-side mmap required:

- `0x42084319` — write SHRAM block (up to 0x200 bytes at a CB-relative offset)
- `0xc2084318` — read SHRAM block (same shape, reverse direction)
- `0x40084304` — write a 32-bit register in cbregs (used to post mailboxes)
- `0x4010431d` — wait on completion with timeout (200 ms cap)

In principle this is a path: open `/dev/ocx_cbregs` (multi-map ok),
write Forward_Open bytes to a CB via `0x42084319`, post the CB to
mbox 0x204 via `0x40084304`, wait via `0x4010431d`, read response
via `0xc2084318`.  No SHRAM mmap needed.

**But we cannot allocate a CB.**  The CB free-list lives inside
SHRAM and is walked by `apx_alloc_cb` / `OC_bpFreeListAlloc` in
libocxbpeng — both of which require a SHRAM mmap.  None of the
exposed ioctls offers a "give me a fresh CB ID" primitive; the
SAM CB queue ioctls (`0x80044309/0a/0b/0c/24`) dequeue from per-type
completion queues, not the free pool, so using them would steal
CBs out from under bpServer's actively-running workers and
deterministically corrupt its state.

Even with the ioctl back-door, the fundamental coexistence
constraint — "we share a CB allocator with bpServer and have no
way to coordinate" — remains a complete blocker.

### Other paths we considered, all blocked

- **Replace bpServer.**  Requires root to kill it + ASEM-shipped
  binary we can't modify.  Out of scope.
- **Solo-mode deployment (stop bpServer first).**  Same root
  requirement.  Plus the HMI's other userland (display update,
  time sync, `labelverification-gw`) depends on bpServer.
- **Fork bpServer or have it spawn us.**  Not possible from a
  sidecar container; bpServer is started by systemd as root and
  we can't inject children.
- **Bypass libocxbpeng and go SHRAM-direct (sibling apex2d
  approach).**  Same SHRAM mmap requirement; same `-EBUSY`.

### Conclusion

**v0.8 large-buffer connected messaging via mbox 0x204 is NOT
POSSIBLE from a non-privileged container on the cm1756 image we
target.**  Not because of a missing symbol or unsolved RE problem
— the path is well understood — but because the operational
constraint (rootless Docker + bpServer is ASEM-owned and runs
continuously) is incompatible with the kernel module's mmap
exclusion model.

This conclusion holds regardless of how much more RE we do.  The
shram exclusion is enforced unconditionally at the kernel layer
and our process cannot satisfy the precondition.

### What v0.8 could still ship (if you want a v0.8 at all)

The v0.7.0 small-buffer TxRx path is fully functional.  v0.8
could ship QoL improvements on top of it:

- **Multi-hop routes** — let callers target devices off-chassis by
  embedding `Unconnected_Send` (svc 0x52) inside the txrx_msg
  request body.  Out-of-scope today (we enforce backplane-direct
  `{0x01, slot}` only); the routing already works through
  MessageSend if the request body carries the embedded path.
- **Connection pool** — pre-open N connections per slot at SDK
  init, hand out from the pool on demand.  Useful for callers
  that do many short conversations against the same PLC.
- **Connection keepalive** — periodic cheap Identity GAA on idle
  conns to prevent the PLC's idle-timeout from invalidating
  them mid-session.  Mirrors the sibling's
  `slot_pool_keepalive_idle` (apex2_cip_connection.c:3287).
- **Async batching** — pipeline multiple TxRxMsg calls through
  the slot dispatcher concurrently from multiple threads (the
  underlying bpServer IPC already supports this — the SDK just
  needs to expose a non-blocking variant or a goroutine /
  thread-pool wrapper).
- **Error-code hardening** — return `BpEngine`-style structured
  errors when the LFO reply carries an extended status (status
  0x02 / ext 0x0100 "connection in use", etc.) instead of
  collapsing to BP_ERR_GENERIC + stderr.

These don't unlock 4002-byte packets but they make v0.7's
small-buffer surface more production-friendly.  Pick whichever
matter for your workload; none are blocked by the SHRAM constraint.

The 4002-byte story is **shelved indefinitely** unless one of the
following changes:

1. ASEM ships a kernel module update that relaxes the shram
   single-mapper exclusion (e.g., reference-counted shared
   mapping, or a privileged-bypass ioctl).
2. bpServer is replaced or extended by ASEM to expose Class3
   over its IPC.
3. The deployment moves to a privileged container model where
   we can stop bpServer and run our SDK as the only userland.
